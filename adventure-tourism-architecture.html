02 — Architecture Strategy

Modular Monolith → Microservices

The biggest startup mistake is over-engineering from day one. The recommended journey is intentional scaling — start simple, design modular, extract services as real demand requires it.

⚠️ Do NOT start with 20 microservices.

Most startups fail because they over-engineer. Begin with a modular monolith for faster development, easier debugging, lower DevOps cost, and faster iteration.

01
Modular Monolith
0 → 100K Users

A single deployable unit with well-defined internal modules. Fast to build, easy to debug, cheap to operate.

Faster development velocity
Easier deployment pipeline
Lower DevOps cost
Simpler ACID transactions
Faster product iteration

04 — Database Architecture

Polyglot Persistence Strategy

A tourism marketplace requires different data engines for different needs. One database cannot serve them all optimally.

Database Technology Primary Use Cases Why This Choice
Primary DB PostgreSQL + PostGIS Bookings, users, tours, vendors, transactions ACID compliance, geospatial queries, JSON support, reliability
Cache Layer Redis (ElastiCache) Session storage, OTP, rate limiting, availability cache Sub-millisecond reads, pub/sub, TTL-based expiry
Search Engine OpenSearch / Elasticsearch Tour search, filters, full-text, location ranking Tourism apps need powerful faceted search and geo-ranking
Geo Extension PostGIS Nearby tours, distance calc, route mapping Native geospatial indexing within PostgreSQL
🗺️ PostGIS is non-negotiable.

For an adventure tourism platform, geospatial queries are first-class citizens. Finding nearby tours, calculating distances, and mapping adventure routes all require PostGIS extensions on PostgreSQL.

10 — DevOps & Observability

Container-First · Observe Everything

Marketplace apps require deep observability. You must know what's failing before your customers do.

Containerisation
Docker
Start with Docker Compose locally, ECS Fargate in production. Move to EKS when orchestration complexity demands it.
CI / CD Pipeline
GitHub Actions
Automated test, build, and deploy. Add ArgoCD for GitOps-style Kubernetes deployments later.
Logging
ELK Stack
Elasticsearch + Logstash + Kibana. Centralised structured logs across all services.
Metrics
Prometheus
Pull-based metrics collection with alerting rules. Pairs naturally with Spring Boot Actuator.
Dashboards
Grafana
Visualise metrics, logs, and traces in a single pane. Set up SLA/SLO dashboards for booking and payment flows.
Tracing & Errors
OpenTelemetry + Sentry
Distributed tracing across services (Zipkin/OTEL) + real-time error tracking (Sentry) for fast incident response.
11 — Security

Security at Every Layer

Tourism platforms process payments and personal data. Security is not optional — it's foundational.

🔒

HTTPS Everywhere

TLS termination at load balancer. All internal service communication encrypted.

🛡️

Web Application Firewall

AWS WAF rules to block SQLi, XSS, and common OWASP Top 10 vectors.

⏱️

Rate Limiting

API Gateway + Redis-backed rate limiting per user, IP, and endpoint.

🍪

Secure Cookies & CSRF

HttpOnly, SameSite=Strict cookies. CSRF tokens on all state-changing requests.

🔑

Secrets Management

AWS Secrets Manager for all credentials. Zero secrets in code or environment variables.

🗄️

SQL Injection Prevention

Parameterised queries via Hibernate/JPA. No raw SQL concatenation anywhere.

13 — Anti-Patterns

Biggest Mistakes to Avoid

The graveyard of failed travel tech startups is littered with beautiful over-engineered architectures that never shipped, or shipped too slowly to compete.

✗ What to Avoid
  • Starting with 20 microservices
  • Deploying Kubernetes on Day 1
  • Using 15 different databases
  • Building custom auth from scratch
  • Over-engineering before product-market fit
  • Ignoring SEO (SSR matters for tourism)
  • Tight coupling to a single payment provider
  • Skipping observability until production breaks
✓ What to Do Instead
  • Start with a modular monolith
  • Use ECS Fargate or simple Docker Compose
  • Use PostgreSQL + Redis to start
  • Use Keycloak or Auth0
  • Design modular, scale intentionally
  • Use Next.js for SSR from Day 1
  • Build a payment abstraction layer
  • Set up Grafana + Prometheus early
